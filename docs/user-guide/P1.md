# Multi-Project Design System MCP — Phase 1 User Guide

Welcome to the **Multi-Project Design System MCP**, the centralized platform for managing design tokens, component specifications, and conformance validation across your organization's product suite.

## Overview

Phase 1 delivers the core foundation of the design system: a registry for managing design system projects, a token store with support for inheritance and overrides, a component specification catalog, and built-in WCAG 2.2 Level AA validation. All capabilities are exposed through a REST API and MCP server suitable for both manual operations and AI-assisted workflows.

### What You Can Do

- **Organize Projects**: Create an organization-wide base design system and let each product team inherit and customize it.
- **Manage Tokens**: Define design tokens (colors, spacing, typography, and more) once in your base system and override them per project.
- **Catalog Components**: Document UI component specs with props, variants, and accessibility rules; inherit and override them per project.
- **Validate WCAG Compliance**: Check color contrast, target size, focus visibility, and ARIA states for accessibility conformance.
- **Export & Integrate**: Resolve merged token sets and export as CSS or JSON for use in your products.

## Quick Start

### Prerequisites

- Node.js 16+
- SQLite 3 (included via Node.js SQLite bindings)
- `curl` or a REST client for manual API calls
- A Bearer token (set via `MCP_SECRET` env var)

### Start the Server

```bash
export MCP_PORT=3100
export MCP_SECRET=your-secure-token-here
export DB_PATH=./data/mpds.db

npm run start
# Server listens on http://localhost:3100
```

### Verify It's Running

```bash
curl http://localhost:3100/health
# Expected: { "status": "ok" }
```

### Make Your First API Call

Create a new base design system project:

```bash
curl -X POST http://localhost:3100/api/projects \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "base-design-system",
    "name": "Base Design System"
  }'
```

Expected response (201 Created):

```json
{
  "id": "base-design-system",
  "name": "Base Design System",
  "parentId": null,
  "createdAt": "2026-06-13T10:00:00Z"
}
```

## Feature 1: Project Registry

The project registry lets you organize design systems hierarchically. Start with a single **base** project (the organization-wide source of truth), then create **child** projects for individual product teams. Child projects inherit all base tokens and component specs but can layer project-specific overrides.

### Hierarchy Rules

- **Base project**: no `parentId`. Your organization has exactly one base.
- **Child project**: `parentId` points to the base project. Each product team gets one child.
- **No grandchildren**: Child projects cannot have children — only two levels of inheritance.
- **Inheritance**: A child project inherits all tokens and specs from its parent; overrides apply only to that child.

### Create the Base Project

```bash
curl -X POST http://localhost:3100/api/projects \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "org-design-base",
    "name": "Organization Design Base"
  }'
```

### Create a Child Project (Product Team)

```bash
curl -X POST http://localhost:3100/api/projects \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "product-mobile-app",
    "name": "Mobile App Design System",
    "parentId": "org-design-base"
  }'
```

### List All Projects

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/projects
```

### Update a Project Name

```bash
curl -X PATCH http://localhost:3100/api/projects/product-mobile-app \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{"name": "Mobile App v2 Design System"}'
```

### Delete a Project

```bash
curl -X DELETE http://localhost:3100/api/projects/product-mobile-app \
  -H "Authorization: Bearer your-secure-token-here"
```

> You cannot delete a base project that still has child projects, or any project that still has tokens attached. Remove dependents first.

### Common Errors

| Situation | Status | Notes |
|-----------|--------|-------|
| Duplicate project ID | 409 | A project with that ID already exists. |
| Project not found | 404 | The referenced project does not exist. |
| Base project has children | 409 | Remove child projects first. |
| Project has tokens attached | 409 | Delete all tokens for the project first. |
| Grandchild creation rejected | 400 | Only one level of inheritance is allowed. |

## Feature 2: Design Tokens

Design tokens are the primitive values of your design system: colors, spacing sizes, font settings, border radii, shadows, and breakpoints. Define them once in your base project and let product teams override specific values locally.

### Token Categories

`color` · `spacing` · `typography` · `radius` · `shadow` · `breakpoint`

### Token Types

- **Primitive token**: Holds a raw value (e.g., `color-primary-blue: #0066cc`).
- **Semantic token**: References another token key, creating an alias (e.g., `color-button-fill` → `color-primary-blue`).

### Create a Primitive Token

```bash
curl -X POST http://localhost:3100/api/tokens \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "org-design-base",
    "key": "color-primary-blue",
    "category": "color",
    "value": "#0066cc"
  }'
```

### Create a Semantic Token

```bash
curl -X POST http://localhost:3100/api/tokens \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "org-design-base",
    "key": "color-button-fill",
    "category": "color",
    "semanticRef": "color-primary-blue"
  }'
```

Semantic tokens resolve to their referenced primitive's value at read time.

### List Tokens (with optional category filter)

```bash
# All tokens
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/tokens/org-design-base

# Only color tokens
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/tokens/org-design-base?category=color
```

### Set a Project-Level Override

Override the primary color in your mobile app without changing the base:

```bash
curl -X PUT http://localhost:3100/api/projects/product-mobile-app/tokens/color-primary-blue/override \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "value": "#0055bb",
    "version": 0
  }'
```

> The `version` field enables **optimistic concurrency control (OCC)**. Supply the current version number — if the token was updated by another client in the meantime, you receive `409 CONFLICT`. Fetch the current version and retry.

### Get the Merged Token Set (Base + Overrides)

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/projects/product-mobile-app/tokens
```

Each entry has a `source` field: `base` (from the base project) or `override` (your project's customization).

### Update a Token (with OCC)

```bash
curl -X PUT http://localhost:3100/api/tokens/org-design-base/color-primary-blue \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "value": "#0077dd",
    "version": 1
  }'
```

### Export Tokens as CSS

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  "http://localhost:3100/api/projects/product-mobile-app/tokens/export?format=css"
```

Output:
```css
:root {
  --color-primary-blue: #0055bb;
  --color-button-fill: #0055bb;
}
```

### Export Tokens as JSON

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  "http://localhost:3100/api/projects/product-mobile-app/tokens/export?format=json"
```

### Delete a Project Override

```bash
curl -X DELETE \
  http://localhost:3100/api/projects/product-mobile-app/tokens/color-primary-blue/override \
  -H "Authorization: Bearer your-secure-token-here"
```

### Common Errors

| Situation | Status | Error Code | Notes |
|-----------|--------|-----------|-------|
| Token not found | 404 | TOKEN_NOT_FOUND | The referenced token does not exist. |
| Invalid category | 400 | INVALID_CATEGORY | Must be one of the six supported categories. |
| Duplicate token key | 409 | DUPLICATE_TOKEN_KEY | A token with that key already exists in this project. |
| Circular reference | 400 | CIRCULAR_REFERENCE | The semantic reference chain loops back to itself. |
| Token is aliased by another | 409 | TOKEN_REFERENCED_BY_SEMANTIC | Delete the semantic token that references this one first. |
| Stale version | 409 | CONFLICT | Another client updated the token. Fetch the current version and retry. |

## Feature 3: Component Specification Catalog

Document your UI components once in the base project, and let product teams override specific fields per project. A component spec captures name, description, props, variants, states, usage rules, and accessibility notes.

### Create a Base Component Spec

```bash
curl -X POST http://localhost:3100/api/components \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "button",
    "projectId": "org-design-base",
    "name": "Button",
    "description": "Primary interactive element",
    "props": [
      {
        "name": "variant",
        "type": "\"primary\" | \"secondary\"",
        "required": true,
        "description": "Button style variant"
      }
    ],
    "variants": ["primary", "secondary"],
    "states": ["default", "hover", "active", "disabled"],
    "usageRules": ["Always provide a label", "Use primary for main actions"],
    "accessibilityNotes": ["Ensure sufficient color contrast", "Support keyboard navigation"]
  }'
```

### Get a Resolved Component Spec

Returns the fully merged spec for a project — base fields plus any project overrides, with per-field source tracking:

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/components/product-mobile-app/button
```

Response includes `_sources` indicating the origin of each field:

```json
{
  "id": "button",
  "name": "Button",
  "variants": ["primary", "secondary", "ghost"],
  "_sources": {
    "name": "base",
    "variants": "override",
    "states": "base",
    "usageRules": "base"
  }
}
```

### List All Specs in a Project

```bash
curl -H "Authorization: Bearer your-secure-token-here" \
  http://localhost:3100/api/components/product-mobile-app
```

### Update a Base Spec (with OCC)

```bash
curl -X PATCH http://localhost:3100/api/components/org-design-base/button \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "Primary interactive element with enhanced interactivity",
    "version": 1
  }'
```

### Set a Per-Project Override

Replace specific array fields for your product team (other fields fall through from the base):

```bash
curl -X POST http://localhost:3100/api/components/product-mobile-app/button/overrides \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "variants": ["primary", "secondary", "ghost"]
  }'
```

> **Array replacement rule**: When you override an array field (`props`, `variants`, `states`, `usageRules`, `accessibilityNotes`), the override entirely replaces the base array — there is no element-level merge.

### Delete an Override

Removes your project's overrides; subsequent calls return the base spec values:

```bash
curl -X DELETE http://localhost:3100/api/components/product-mobile-app/button/overrides \
  -H "Authorization: Bearer your-secure-token-here"
```

### Delete a Component Spec

Deletes the base spec and cascades removal of all per-project overrides:

```bash
curl -X DELETE http://localhost:3100/api/components/org-design-base/button \
  -H "Authorization: Bearer your-secure-token-here"
```

### Common Errors

| Situation | Status | Error Code | Notes |
|-----------|--------|-----------|-------|
| Component not found | 404 | COMPONENT_NOT_FOUND | No spec with that ID exists. |
| Duplicate component ID | 409 | DUPLICATE_COMPONENT_ID | A spec with that ID already exists. |
| Invalid override field | 400 | INVALID_OVERRIDE_FIELD | Only description, props, variants, states, usageRules, accessibilityNotes can be overridden. |
| Stale version | 409 | CONFLICT | Another client updated the spec. Fetch the current version and retry. |
| Project not found | 404 | PROJECT_NOT_FOUND | The referenced project does not exist. |

## Feature 4: WCAG 2.2 Conformance Validation

Check color contrast, interactive target size, focus visibility, and ARIA markup against WCAG 2.2 Level AA. This is a pure computation engine — no database writes, safe to call freely.

### What Gets Checked

| Criterion | Rule | Level |
|-----------|------|-------|
| 1.4.3 Contrast Minimum | Normal text ≥ 4.5:1; large text & UI elements ≥ 3:1 | AA |
| 2.5.8 Target Size | Clickable targets ≥ 24×24 px | AA |
| 2.4.11 Focus Visible | Focus indicator must be visible (warning) | AA |
| 4.1.2 ARIA States | ARIA attributes must be present and valid | AA |

### Accepted Color Formats

`#RGB` · `#RRGGBB` · `rgb(R,G,B)`

### Validate a Color Pair

```bash
curl -X POST http://localhost:3100/api/validate/color-pair \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "foreground": "#000000",
    "background": "#ffffff",
    "context": "normal"
  }'
```

Valid contexts: `normal` (body text), `large` (18 px+ or 14 px bold), `ui-element`.

Response:
```json
{
  "ratio": 21.0,
  "passes": { "normal": true, "large": true, "ui-element": true }
}
```

### Reference Contrast Values

| Pair | Ratio | Normal (4.5:1) | Large / UI (3:1) |
|------|-------|---------------|-----------------|
| `#000` / `#fff` | 21.0 | ✓ | ✓ |
| `#767676` / `#fff` | 4.54 | ✓ | ✓ |
| `#777` / `#fff` | 4.48 | ✗ | ✓ |

### Validate an HTML/JSX Snippet

Extracts colors from inline `style=` attributes and validates them:

```bash
curl -X POST http://localhost:3100/api/validate/snippet \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "input": "<button style=\"color: #000; background: #fff;\">Click me</button>"
  }'
```

### Validate Token Usage

Check if two tokens from your design system form an accessible color pair:

```bash
curl -X POST http://localhost:3100/api/validate/token-pair \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "product-mobile-app",
    "fgToken": "color-primary-blue",
    "bgToken": "color-background",
    "context": "normal"
  }'
```

### Common Errors

| Situation | Status | Error Code | Notes |
|-----------|--------|-----------|-------|
| Invalid color format | 400 | INVALID_COLOR_FORMAT | Must be `#RGB`, `#RRGGBB`, or `rgb(R,G,B)`. |
| Token is not a color | 404 | TOKEN_NOT_A_COLOR | Referenced token does not exist or is not a color category. |

## Feature 5: MCP Server for AI Assistants

In addition to the REST API, the server exposes an MCP (Model Context Protocol) endpoint for AI assistants and automated workflows.

**Endpoint**: `POST http://localhost:3100/mcp`

### Available MCP Tools

| Tool | Description |
|------|-------------|
| `list_projects` | List all design system projects |
| `get_design_system` | Get all resolved tokens and component specs for a project in one call |
| `get_tokens` | Get resolved tokens for a project (optionally filtered by category) |
| `get_component_spec` | Get a resolved component spec by project and component ID |
| `validate_color_pair` | Validate foreground/background contrast |
| `validate_token_pair` | Validate a token pair for contrast |
| `validate_snippet` | Validate an HTML/JSX snippet for WCAG conformance |

### Example MCP Call

```bash
curl -X POST http://localhost:3100/mcp \
  -H "Authorization: Bearer your-secure-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
      "name": "get_design_system",
      "arguments": { "projectId": "product-mobile-app" }
    }
  }'
```

## Feature 6: Web UI Override Editor

A React-based web editor served at `http://localhost:3100`. No separate install — it comes bundled with the server.

### What You Can Do

- **Project Selector**: Switch between projects.
- **Token Browser**: Browse tokens with live color swatches, create and delete tokens, set per-project overrides with live preview (debounced 300 ms).
- **Component Browser**: Browse and edit component specs; manage per-project overrides.
- **Inline WCAG Feedback**: Contrast badges update automatically as you edit color values.

### Conflict Handling

The UI uses optimistic concurrency control. If another user saved the same token while you were editing, a conflict warning appears. Refresh the token to load the latest values and reapply your change.

## Configuration Reference

| Variable | Type | Default | Required | Description |
|----------|------|---------|----------|-------------|
| `MCP_PORT` | number | `3100` | No | Port the HTTP server listens on. |
| `MCP_SECRET` | string | — | In production | Bearer token for API authentication. |
| `MPDS_ENV` | string | — | No | Set to `production` to require `MCP_SECRET` at startup. |
| `DB_PATH` | string | `./data/mpds.db` | No | Path to the SQLite database file (created automatically). |

### Local Development

```bash
export MCP_PORT=3100
export MCP_SECRET=dev-secret-change-me
export DB_PATH=./data/mpds.db
npm run start
```

### Production

```bash
export MCP_PORT=3100
export MCP_SECRET=$(openssl rand -hex 32)
export DB_PATH=/var/lib/mpds/mpds.db
export MPDS_ENV=production
npm run start
```

In `production` mode the server refuses to start if `MCP_SECRET` is unset, logging:
`FATAL: MCP_SECRET must be set in production mode`

## Troubleshooting

### Server Won't Start — Port Already in Use

```bash
lsof -i :3100          # find the process
kill -9 <PID>           # stop it
# or set MCP_PORT to a free port
```

### 401 Unauthorized on Every Call

Verify the `Authorization` header format:

```bash
curl -H "Authorization: Bearer your-secret" http://localhost:3100/api/projects
```

| Error Code | Cause |
|-----------|-------|
| MISSING_AUTH_HEADER | No `Authorization` header was sent. |
| INVALID_TOKEN | The token doesn't match `MCP_SECRET`. |
| INVALID_AUTH_SCHEME | Used `Basic` instead of `Bearer`. |

### Database Errors

Ensure the directory for `DB_PATH` exists and is writable:

```bash
mkdir -p $(dirname "$DB_PATH")
```

If the DB is corrupted, back it up and remove it — the server recreates the schema on next start:

```bash
mv ./data/mpds.db ./data/mpds.db.bak
npm run start
```

### Circular Reference on Token Creation

Token aliases must not form loops. Example of an invalid chain: `A → B → A`. Review the `semanticRef` fields in your token definitions and remove the cycle.

### 409 Conflict When Saving

Another client updated the resource after you last fetched it. Fetch the current record, read its `version`, and retry your update with that version:

```bash
# 1. Fetch current state
curl -H "Authorization: Bearer $MCP_SECRET" \
  http://localhost:3100/api/tokens/org-design-base/color-primary-blue

# 2. Retry with the version from step 1
curl -X PUT http://localhost:3100/api/tokens/org-design-base/color-primary-blue \
  -H "Authorization: Bearer $MCP_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"value": "#0077dd", "version": <current-version>}'
```

### INVALID_COLOR_FORMAT

Accepted formats only: `#RGB`, `#RRGGBB`, `rgb(R,G,B)`. Values like `rgba(...)`, named colors (`red`), or HSL are not supported in Phase 1.
